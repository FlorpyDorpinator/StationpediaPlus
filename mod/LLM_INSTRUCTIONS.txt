================================================================================
STATIONPEDIA ASCENDED - LLM INSTRUCTIONS FOR WRITING descriptions.json ENTRIES
================================================================================

You are helping write device documentation for the Stationeers game mod 
"Stationpedia Ascended". This mod enhances the in-game encyclopedia with 
custom tooltips and descriptions.

--------------------------------------------------------------------------------
FILE STRUCTURE
--------------------------------------------------------------------------------

The descriptions.json file has this top-level structure:

{
  "version": "1.0",
  "devices": [ ... array of device entries ... ],
  "genericDescriptions": { ... global fallback descriptions ... }
}

--------------------------------------------------------------------------------
DEVICE ENTRY FORMAT
--------------------------------------------------------------------------------

Each device in the "devices" array follows this structure:

{
  "deviceKey": "ThingStructureDeviceName",    // REQUIRED - exact prefab name
  "displayName": "Human Readable Name",        // Optional - for documentation only
  
  // Page description options (pick ONE or combine append/prepend):
  "pageDescription": "Replaces entire description",
  "pageDescriptionAppend": "Added to end of existing description",
  "pageDescriptionPrepend": "Added to beginning of existing description",
  
  // Operational Details - MUST BE AN ARRAY:
  "OperationalDetails": [
    {
      "title": "Section Title",
      "description": "Section content goes here.",
      "items": ["Bullet item 1", "Bullet item 2"],
      "steps": ["Step one", "Step two", "Step three"],
      "children": [ /* nested OperationalDetails */ ]
    }
  ],
  "operationalDetailsTitleColor": "#FF7A18",   // Optional hex color
  
  // Logic tooltips - object with logic type names as keys:
  "logicDescriptions": {
    "Temperature": {
      "dataType": "Float",
      "range": "0+",
      "description": "Internal temperature in Kelvin."
    }
  },
  
  // Mode tooltips - object with mode names as keys:
  "modeDescriptions": {
    "Mode0": {
      "modeValue": "0",
      "description": "What mode 0 does."
    }
  },
  
  // Slot tooltips:
  "slotDescriptions": {
    "InputSlot": {
      "slotType": "Ore",
      "description": "Accepts ore for processing."
    }
  }
}

--------------------------------------------------------------------------------
OPERATIONALDETAILS - SUPPORTED PROPERTIES
--------------------------------------------------------------------------------

Each item in the OperationalDetails array can have:

  "title"       - (string) Section heading, displayed in orange
  "description" - (string) Main text content
  "items"       - (array of strings) Bullet list, rendered with • prefix
  "steps"       - (array of strings) Numbered list, rendered as 1. 2. 3.
  "children"    - (array) Nested OperationalDetails for sub-sections

All properties are OPTIONAL. Use what makes sense for your content.

--------------------------------------------------------------------------------
NESTING EXAMPLE
--------------------------------------------------------------------------------

"OperationalDetails": [
  {
    "title": "Steel Smelting",
    "description": "The most common furnace recipe.",
    "children": [
      {
        "title": "Requirements",
        "items": [
          "Iron Ore + Coal Ore in 3:1 ratio",

          "Pressure above 100 kPa"
        ]

          "Close the furnace door",

          "Wait for Mode=1 or temperature 900K+",
          "Collect ingots from export slot"
        ]

    "items": [
      "Furnace handles extreme pressures safely",
      "On cold planets, use more fuel ice",

This renders as:

  Steel Smelting
  The most common furnace recipe.

      • Iron Ore + Coal Ore in 3:1 ratio
      • Temperature above 900K
      • Pressure above 100 kPa
    
    Process

    • Furnace handles extreme pressures safely

    • Lock furnace via logic to prevent accidents

--------------------------------------------------------------------------------
TEXTMESHPRO RICH TEXT FORMATTING
--------------------------------------------------------------------------------

The mod uses TextMeshPro for rendering. These tags are supported:

BASIC FORMATTING:
  <b>bold text</b>
  <i>italic text</i>
  <u>underlined text</u>
  <s>strikethrough</s>

COLORS:
  <color=#FF0000>red text</color>
  <color=#00FF00>green text</color>
  <color=#FFA500>orange text</color>
  <color=red>named color</color>

SIZE:
  <size=150%>bigger text</size>
  <size=80%>smaller text</size>
  <size=24>24pt text</size>

ALIGNMENT:
  <align=center>centered text</align>
  <align=right>right-aligned</align>

INDENTATION:
  <indent=10%>indented text</indent>
  <indent=2em>indented by 2em</indent>

SUPER/SUBSCRIPT:
  H<sub>2</sub>O  →  H₂O
  x<sup>2</sup>   →  x²

OTHER:
  <nobr>prevent line break</nobr>
  <lowercase>MAKES LOWERCASE</lowercase>
  <uppercase>makes uppercase</uppercase>
  <smallcaps>Small Caps Text</smallcaps>

LINE BREAKS:
  Use \n in JSON strings for line breaks.

--------------------------------------------------------------------------------
BULLET AND LIST CHARACTERS
--------------------------------------------------------------------------------

When writing inline lists (in description strings), use these Unicode chars:

  • (U+2022) - standard bullet
  ‣ (U+2023) - triangular bullet
  ○ (U+25CB) - hollow circle
  ► (U+25BA) - triangle pointer
  → (U+2192) - arrow

Example in description:
  "description": "• First point\n• Second point\n• Third point"

NOTE: Prefer using the "items" or "steps" arrays instead of manual bullets
in the description - they render more consistently.

--------------------------------------------------------------------------------
CRITICAL RULES
--------------------------------------------------------------------------------

1. OperationalDetails MUST be an ARRAY (square brackets [])
   
   WRONG:  "OperationalDetails": { "title": "..." }
   CORRECT: "OperationalDetails": [ { "title": "..." } ]

2. Use PascalCase "OperationalDetails" (capital O and D)

3. All properties in each OperationalDetails item are OPTIONAL
   - You can have just "title"
   - You can have just "items" 
   - You can have any combination

4. Nesting depth: Keep it reasonable (2-3 levels max for readability)

5. JSON escaping: Use \\n for newlines, \\" for quotes
